# Silicon -- A modern hardware language

An experimental hardware description language and compiler, built on top of CIRCT, MLIR, and LLVM.
The goal is to create a modern language with an ergonomic syntax inspired by Rust and Zig, which allows for rich metaprogramming in the language itself, rather than using a separate macro or template system put ontop the language.
A large part of hardware engineering is fine-tuning the precise structure of the expression and operations, similar to how a regex compiler or instruction selection table generator generates carefully structured code to achieve the best performance.
Silicon's metaprogramming is based on phased execution, where code can be marked as constant or dynamic, indicating that it should run at an earlier or later phase of compilation.
This is similar to how the quoting and unquoting works in Lisp or Scala 3 macros.
Hardware is generated by executing code in all but the final phase, and mapping the remaining IR to CIRCT's core dialects.

## Compilation Flow

**Parse the input.**
The lexer splits the input into tokens, which the parser then uses to build an AST.
The AST is a simple tree structure that closely follows the syntax of the source code.

**Conversion to HIR.**
The code generator then converts the AST into the HIR dialect.
In this dialect, types are SSA values and type constructors are regular ops, instead of being `mlir::Type`s.

**Lower earliest phase to MIR.**
Convert the earliest phase of code to the MIR dialect.
This requires looking at SSA values used as type operands on ops, enforcing that they are a materialized constant, and using that constant as `mlir::Type` of the corresponding MIR op.
MIR ops closely follow HIR ops, but have types and constants expressed as `mlir::Type` and `mlir::Attribute`, instead of SSA operands.

**Execute earliest phase.**
Execute the earliest phase of the code, which was lowered to MIR in the previous step.
This computes constant values and types, and materializes them as constant ops in the IR, feeding the next phase of the code.
Go back to the previous step and repeat until all but the last phase has been executed.

**Lower to CIRCT.**
Lower the final phase of code to CIRCT's core dialects, which can then be further optimized, simulated, synthesized, or translated to Verilog.

## Intermediate Representation

The split into an HIR and MIR dialect is the language's main mechanism for metaprogramming.
HIR ops carry types as SSA operands, while MIR ops have them materialized as regular MLIR types:

```mlir
%0 = hir.int_type
hir.add %a, %b : %0
%1 = hir.constant_int 42
%2 = hir.array_type %0, %1
hir.array_get %c[%d] : %2, %0
// equivalent to
mir.add %a, %b : !mir.int
mir.array_get %c[%d] : !mir.array<42 x !mir.int>, !mir.int
```

Carrying types as SSA values allows the IR to represent type and constant inference as a dedicated op:

```mlir
%0 = hir.inferrable
%1 = hir.constant_int 42 : %0
%2 = hir.int_type
%3 = hir.variable : %2
%4 = hir.unify %0, %2  // assignment requires LHS and RHS types to match
hir.assign %3, %1 : %4
```

## Phase Split

The `const` and `dyn` keywords are used to move expressions, types, and other code to an earlier or later phase of execution relative to the surrounding code, respectively.
For example:

```silicon
let a = 42;  // executed in phase 0; a is the constant 42 in phase 1
let b = const { 1337 };  // executed in phase -1; b is the constant 1337 in phase 0
let c = dyn { a + b };  // executed in phase 1; c is the constant 1379 in phase 1
let d = const {  // executed in phase -1
  let e = const { 9001 };  // executed in phase -2; e is the constant 9001 in phase -1
  let f = b + e;  // executed in phase -1; f is the constant 10338 in phase 0
  let g = dyn { a };  // executed in phase 0; g is the constant 42 in phase 0
  // g is unknown/symbolic in phase -1; it can be used in exprs, but makes the expression symbolic:
  let h = g + 1;  // executed in phase 0; h is the symbolic expr `42 + 1` in phase -1; constant 43 in phase 0
  // symbolic values cannot be used for control flow:
  if h > 40 { ... }  // ILLEGAL since h is phase 0, but if is executed in phase -1
  dyn { if h > 40 { ... } }  // OK since the if is now executed in phase 0
};
```

Function arguments and results can also be marked `const` or `dyn`, to indicate in which phase they are evaluated relative to the function's call site.
This is the main specialization mechanism in the language, allowing for constant arguments and results to be computed in an earlier phase and then inlined into the function body.
The inlining leads to monomorphization of the function.
For example:

```silicon
fn foo(
  const a: int,
  b: int,
  dyn c: int,
) -> (
  const d: int,
  e: int,
  dyn f: int
) { ... }

let (d, e, f) = foo(a, b, c);  // executed in phase 0; a/d must be phase -1, b/e must be phase 0; c/f must be phase 1
let (d, e, f) = const { foo(a, b, c); }  // executed in phase -1; a/d must be phase -2, b/e must be phase -1, c/f must be phase 0
```

The entire function can be marked with one or more `const` or `dyn`, to shift the phase of the arguments, results, and body relative to the surrounding code:

```silicon
const fn foo(a: int) -> (b: int) { ... }
dyn fn bar(a: int) -> (b: int) { ... }
// is equivalent to
fn foo(const a: int) -> (const b: int) { const { ... } }
fn bar(dyn a: int) -> (dyn b: int) { dyn { ... } }
```

It is an error to use a value from a later phase in an earlier phase, since the value is unknown or symbolic at that point.
The exception are side-effect free expressions such as `+`, which determine their phase as the latest of their operands.
Potentially side-effecting operations such as `if` control flow, or function calls, determine their phase based on the surrounding code and force operands to be in the correct phase.
For example:

```silicon
let a = dyn { dyn { 42 } };  // executed in phase 2
let b = dyn { 1337 };  // executed in phase 1
let c = a + b;  // OK since side-effect free; executed in phase 2, max of a and b phases
if a == 0 { ... }  // ILLEGAL since a only known in phase 2, but if executed in phase 0
const {
  // const body executes in phase -1
  a + b  // OK; executed in phase 2 as determined by a and b phases
};
```

### IR

Converting the AST to HIR generates unified functions that have arguments, results, and body expressions of all phases contained in one function.
An important first step after converting the AST to HIR is to split functions and expressions into phases that live in distinct IR ops and regions.
For example:

```silicon
fn foo(a: int) -> (z: int) {
  let (c, d) = bar(40, a);  // c is phase 0, d is phase 1
  let (e, _) = bar(d + 1, a);  // e is phase 0
  c + e
}
fn bar(const x: int, y: int) -> (u: int, const v: int) {
  let i = const { x + 2 };
  (y * i, i)
}
```

Parsing this input and converting the AST to HIR yields something that looks like this:

```mlir
hir.unified_func @foo(a: 0) -> (z: 0) {  // 0 indicates the argument/result phase
^bb0(%a: !hir.any):
  // constant-like ops have phase -inf
  %0 = hir.int_type  // phase -1 since used as type of argument in phase 0
  hir.unified_signature (%0) -> (%0)
} {
^bb0(%a: !hir.any):
  %0 = hir.constant_int 40  // phase -inf since const-like; otherwise -1 since used for arg in phase -1
  %c, %d = hir.unified_call @bar(%0, %a) : (-1, 0) -> (0, -1)
  %1 = hir.constant_int 1  // phase -inf since const-like; otherwise -1 since used for arg in phase -1
  %2 = hir.phase_shift -1 {  // execute `d + 1` in phase -1 since used for arg in phase -1
    %3 = hir.add %d, %1
    hir.yield %3
  }
  %e, %_ = hir.unified_call @bar(%2, %a) : (-1, 0) -> (0, -1)
  %4 = hir.add %c, %e  // phase 0 since result used for result in phase 0; operands must be phase <=0
  hir.unified_return %4
}
hir.unified_func @bar(x: -1, y: 0) -> (u: 0, v: -1) {
^bb0(%x: !hir.any, %y: !hir.any):
  %0 = hir.int_type  // phase -2 since used as type of argument in phase -1
  hir.unified_signature (%0, %0) -> (%0, %0)
} {
^bb0(%x: !hir.any, %y: !hir.any):
  %0 = hir.constant_int 2  // phase -inf since const-like, otherwise -1 since used for result in phase -1
  %i = hir.phase_shift -1 {  // execute `x + 2` in phase -1 since used for result in phase -1
    %1 = hir.add %x, %0
    hir.yield %1
  }
  %2 = hir.mul %y, %i  // phase 0 since used for result in phase 0; operands must be phase <=0
  hir.unified_return %2, %i
}
```

After phase splitting the IR should look something like the following.
Note that the IR is kept verbose for illustrative purposes.
In practice, we would try to unify types as much as possible early on, and we would not carry constant types like `hir.int_type` between phases of a function.

```mlir
// A split_func captures how a unified function got separated into phases.
// This allows us to split and execute as many phases as possible when compiling a library, but still allow unified calls to split funcs.
hir.split_func @foo(a: 0) -> (z: 0) {
  // This is just a sketch.
  // Each split here needs to provide enough information to know how many additional constants are returned that need to be fed into the next split.
  [-2]: @foo.split0  // returns types of bar.x/v/i
  [-1]: @foo.split1  // takes types of bar.x/v/i, returns bar1.i/bar2.i and types of a/z and bar.i/y/u
  [0]: @foo.split2   // takes bar1.i/bar2.i/a and types of a/z and bar.i/y/u, returns z
}
hir.func @foo.split0() -> (bar.x.type, bar.v.type, bar.i.type) {
  %0 = hir.type_type
  %bar.x.type, %bar.v.type, %bar.i.type = hir.call @bar.split0() : () -> (%0, %0, %0)
  %_, %_, %_ = hir.call @bar.split0() : () -> (%0, %0, %0)  // in theory we'd have to thread these results through, too
  hir.return %bar.x.type, %bar.v.type, %bar.i.type : %0, %0, %0
}
hir.func @foo.split1(%bar.x.type, %bar.v.type, %bar.i.type) -> (bar1.i, bar2.i, bar.i.type, bar.y.type, bar.u.type) {
  %0 = hir.constant_int 40 : %bar.x.type
  %1 = hir.type_type
  %d, %bar1.i, %bar.y.type, %bar.u.type = hir.call @bar.split1(%0, %bar.x.type, %bar.v.type, %bar.i.type) : (%bar.x.type, %1, %1, %1) -> (%bar.v.type, %bar.i.type, %1, %1)
  %2 = hir.constant_int 1 : %bar.v.type
  %3 = hir.add %d, %2 : %bar.v.type
  %_, %bar2.i, %_, %_ = hir.call @bar.split1(%3, %bar.x.type, %bar.v.type, %bar.i.type) : (%bar.x.type, %1, %1, %1) -> (%bar.v.type, %bar.i.type, %1, %1)
  %4 = hir.int_type
  hir.return %bar1.i, %bar2.i, %4, %4, %bar.i.type, %bar.y.type, %bar.u.type : %bar.i.type, %bar.i.type, %1, %1, %1, %1, %1
}
hir.func @foo.split2(%bar1.i, %bar2.i, %a, %a.type, %z.type, %bar.i.type, %bar.y.type, %bar.u.type) -> (z) {
  %0 = hir.type_type
  %c = hir.call @bar.split2(%bar1.i, %a, %bar.i.type, %bar.y.type, %bar.u.type) : (%bar.i.type, %bar.y.type, %0, %0, %0) -> (%bar.u.type)
  %e = hir.call @bar.split2(%bar2.i, %a, %bar.i.type, %bar.y.type, %bar.u.type) : (%bar.i.type, %bar.y.type, %0, %0, %0) -> (%bar.u.type)
  %1 = hir.add %c, %e : %bar.u.type
  hir.return %1 : %bar.u.type
}
hir.split_func @bar(x: -1, y: 0) -> (u: 0, v: -1) {
  [-2]: @bar.split0  // returns types of x/v/i
  [-1]: @bar.split1  // takes x and types of x/v/i, returns v/i and types of y/u
  [0]: @bar.split2   // takes i/y and types of i/y/u, returns u
}
hir.func @bar.split0() -> (x.type, v.type, i.type) {
  %0 = hir.int_type
  %1 = hir.type_type
  hir.return %0, %0 : %1, %1
}
hir.func @bar.split1(%x, %x.type, %v.type, %i.type) -> (v, i, y.type, u.type) {
  %0 = hir.int_type
  %1 = hir.constant_int 2 : %x.type
  %2 = hir.add %x, %1 : %v.type  // probably needs to unify incoming types
  %3 = hir.type_type
  hir.return %2, %2, %0, %0 : %0, %0, %3, %3
}
hir.func @bar.split2(%i, %y, %i.type, %y.type, %u.type) -> (u) {
  %0 = hir.unify %y.type, %i.type
  %1 = hir.unify %0, %u.type
  %2 = hir.mul %y, %i : %1
  hir.return %2 : %1
}
```

As we execute phases of the code, `hir.call` ops get converted into `mir.call` ops to monomorphized versions of the called functions, based on the type and constant arguments provided.
