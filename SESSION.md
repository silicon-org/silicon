- MLIR format caveat: `$variadic` in a custom assembly format fails for empty lists if the variadic is not surrounded by a delimiter like `(...)`, because the internal `parseCommaSeparatedList` with `Delimiter::None` tries to parse at least one element. The fix is to put the variadic inside a delimiter, like `` `(` $variadic `)` ``.
- MLIR `functional-type($args, $results)` prints single results without parentheses (e.g., `(!hir.any) -> !hir.any` not `(!hir.any) -> (!hir.any)`).
- When CheckCalls inlines signature ops before a call, the inlined ops appear after any inferrable placeholders created by codegen. When an inferrable has multiple uses (e.g., both as a call's typeOfArg and as constant_int's type operand), CheckCalls cannot simply RAUW it with the inlined type — this breaks dominance for uses before the inlined ops. The fix is to create a `hir.unify` before the call (where both values are available) and let InferTypes clone the concrete type to the inferrable's position.
- SplitPhases block arg ordering invariant: each split function's block args are ordered as [own args, threaded args from earlier phases]. The const arg handling creates own args in their origin phase, and cross-phase value threading handles forwarding to later phases.
- Unified call decomposition is now done inside `PhaseSplitter::run()` (during splitting), not as a post-split pass. This requires topological processing order so callee split functions exist when decomposing a caller.
- When a single SSA value (e.g., `hir.int_type`) is used as a type operand at multiple phases, the phase assignment must use `std::min` to place it at the earliest needed phase. Value threading then forwards it to later phases.
- Pure zero-operand ops (constants, type constructors) are cloned into each phase during value threading instead of being threaded as block args. This is needed because the HIR-to-MIR lowering requires type operands to be constant ops, not block arguments.
- The interpret pass now chains multi-phase callers: after executing a `.constN` function, it looks for `.const{N-1}` and replaces its block args with the materialized constant return values, making it eligible for execution.
- When decomposing unified calls, `returnValues` in the splitter's auxiliary data structure holds `Value` references that are NOT IR uses. `replaceAllUsesWith` won't update them, so they must be manually updated before erasing the old call op.
- `hir.unify` was missing the `Pure` trait, causing it to be assigned its parent's phase (e.g., 0) instead of being computed from its operands. This dragged binary ops to phase 0 when they should have been in earlier phases. Fixed by adding `Pure`.
- MLIR ODS: `AttrSizedOperandSegments` and `SameVariadicOperandSize` are mutually exclusive. Use `AttrSizedOperandSegments` with a custom verifier instead.
- MLIR ODS: naming a variadic operand `$operands` conflicts with the base `Operation::getOperands()` accessor. Use a different name like `$values`.
- When adding type operands to return ops, the `type_of` values get threaded across phase boundaries along with the data values, increasing the size of opaque packs. These extra type_of ops become dead code in the destination phase but are harmless.
- SplitPhases now inserts `hir.coerce_type` ops on every own block arg in each phase function, annotating them with their type from the unified func's signature. Type values that are signature block args get resolved via the `unifiedArgValues` mapping; type values that are signature op results (like `hir.int_type`) get cloned into the phase function. Cross-phase type operands are handled naturally by the existing value threading loop.
- The strict `shouldLower` check in HIR-to-MIR rejects split-phase functions whose `hir.return.typeOfValues` or `hir.call.typeOfResults` come from `type_of` or `inferrable` ops. These functions produce opaque packs for cross-phase transfer, so the return type metadata is non-constant. Earlier passes need to produce resolvable type operands for these; see TODO.md.
- SplitPhases now inserts `hir.opaque_type` + `hir.coerce_type` on the opaque context block arg, followed by `hir.opaque_unpack` on the coerced value. This gives HIR-to-MIR the type information needed for the context arg. Without it, `FuncOpConversion` couldn't determine the block arg's MIR type.
- The `SymbolTable` analysis in PhaseEvalLoop gets invalidated by the sub-pipeline. Must construct a fresh `SymbolTable` each time instead of using `getAnalysis<SymbolTable>()` across iterations.
- MLIR `ArrayAttr` printing includes type annotations on each element (e.g., `#mir.int<5> : !mir.int` not just `#mir.int<5>`). This affects CHECK lines for `evaluated_func`.
- `SmallPtrSet` cannot be used with MLIR op types (no `PointerLikeTypeTraits` specialization). Use `SmallVector` + `is_contained` instead.
- Use a custom parser/printer like `custom<SymbolVisibility>($sym_visibility)` for visibility keywords (`private`, `nested`).
- SplitPhases now uses a general phase-grouping algorithm: external phases (from arg/result phases) define group boundaries. Internal phases accumulate into the next external phase's group. Multi-phase groups get a `multiphase_func`; standalone groups map directly to `FuncOp`. The old `.const<N>` / `.dyn<N>` naming is replaced by `.<groupIdx>` (standalone) and `.<groupIdx><letter>` (multi-phase).
- When SplitPhases erases a `unified_func` and creates a `split_func` with the same name, the `SymbolTable` must be explicitly updated: `symbolTable.erase(funcOp)` then `symbolTable.insert(splitFuncOp)`. Without this, callers processed later can't find the callee's `split_func`.
- `UnifiedCallOp` verifier now accepts both `unified_func` and `split_func` callees, since SplitPhases processes functions in topological order and a callee may already be split when the caller is verified.
- SplitPhases arg movement was only handling const args (phase < 0); dyn args (phase > 0) stayed in the phase-0 block, causing a vector assertion failure. Fixed by generalizing to handle all non-zero-phase args.
- SpecializeFuncs assumed evaluated functions always produce exactly 1 opaque pack result. For `dyn fn` / `const fn` where there's no cross-phase context, the evaluated function has 0 results. Added early handling to dissolve the multiphase_func without trying to chain results.
- Pre-existing bug: unused function args lose their `coerce_type` during canonicalization/DCE, causing HIR-to-MIR to fail with "block argument type could not be determined". This affects all functions, not just `const fn`/`dyn fn`.
- `hir.if` / `mir.if` have been removed in favor of CFG blocks with `cf.cond_br`/`cf.br` and block arguments. `hir.coerce_to_i1` and `mir.bool_to_i1` bridge the bool-to-i1 gap for branch conditions. MLIR's canonicalizer may collapse trivial CFG patterns into `arith.select`, so HIRToMIR and the interpreter must handle that. When codegen creates nested if/logical-op CFG blocks, merge blocks must be pushed to the region LAST to ensure the function's final block always has the `hir.return` terminator (the verifier checks `getBody().back().getTerminator()`).
- `mod` keyword added: identical syntax to `fn`, carries `isModule` UnitAttr. Only the last phase function (maxPhase) of a module gets `isModule`. The Interpret pass skips `isModule` functions and any function that transitively calls one, preserving module bodies as hardware in the IR.
- SplitPhases now has phase back-propagation: when a call requires an arg at an earlier phase but the value is in an ExprOp at a later phase, `pullPhases` attempts to pull the ExprOp (and transitive deps) to the required phase. This works when all operands/captures are floating constants. Nested `unified_call` arg/result phases must be shifted by the pull delta, and `refreshPhases` re-propagates pulled phases to dependent pure ops (like `type_of`, `unify`). The `maxResultPhase` init was changed from 0 to INT16_MIN to support negative result phases after shifting.
- During HIR-to-MIR dialect conversion, `resolveHIRType` must look through `unrealized_conversion_cast` ops. When the conversion framework converts `hir.bool_type` (or `hir.int_type`) before processing other ops, the type operands get remapped to `mir.constant → unrealized_conversion_cast` chains. `getTypeOf` finds these cast values, which `resolveHIRType` originally couldn't handle. The fix: `resolveHIRType` now looks through single-operand `unrealized_conversion_cast` ops.
- HIRToMIR type converter must pass through builtin types (`IntegerType`, `IndexType`, `FloatType`) in addition to Base/MIR dialect types and `FunctionType`, otherwise `arith.select` with `i1` condition can't be converted.
- Pre-existing doc compilation failures in `const-and-dyn-blocks.md`: `const { 2 + 3 } + x` fails because `hir.unify` survives with different operands after phase eval; `x + dyn { 1 }` fails because the return value is at phase 1 but the function declares phase 0 return. Changed code block tags to `silicon-todo` to skip compilation.
- When `constant_int` has an inferrable type operand, `getOrCreateTypeOf` must NOT return the raw inferrable. Otherwise the inferrable takes on whatever type context provides (even `unit`), silently suppressing type errors. The fix: when `getTypeOf` returns an inferrable, fall through to create `int_type` as the default type for integer literals.
- `InferrableOp` must NOT have the `Pure` trait, because CSE merges distinct inferrables (no operands, no attrs, same result type). Instead, callsites that need to treat it as side-effect-free use `hir::isEffectivelyPure` (defined in `HIR/Ops.h`).
- `const const fn` (double phase shift) fix: the root cause was SpecializeFuncs erasing `evaluated_func` ops (with empty results) while other functions still had void calls to them. The calls are no-ops but weren't cleaned up, leaving dangling symbol references that the interpreter couldn't resolve. Fixed by adding `eraseVoidCalls` that removes all void calls to a symbol before erasing it. The deeper question (TODO) is why SplitPhases generates these void calls in the first place — they represent "run this phase" but the phase produces nothing.
